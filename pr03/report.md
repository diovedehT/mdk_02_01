
ПР №3. Права доступа Linux и управление пользователями

=== 1. Пользователи и группы ===

Созданные пользователи и их роли:
- alice | группа: developers | роль: Администратор проекта
- bob   | группа: developers | роль: Разработчик
- carol | группа: auditors   | роль: Аудитор

--- Поля /etc/passwd ---
Строка: alice:x:1001:1001::/home/alice:/bin/bash
1. alice       — имя пользователя (логин)
2. x           — пароль хранится в /etc/shadow
3. 1001        — UID (уникальный числовой идентификатор)
4. 1001        — GID (идентификатор основной группы)
5. (пусто)     — GECOS (комментарий: имя, контакты)
6. /home/alice — домашняя директория
7. /bin/bash   — командная оболочка (shell)

--- Файл /etc/shadow ---
Хранит хэши паролей, дату смены пароля, срок действия и политику блокировки.
Недоступен обычным пользователям (права -rw-r-----, владелец root),
потому что хэши можно взломать перебором — доступ для всех создал бы угрозу безопасности.


=== 2. Права доступа chmod/chown ===

Итоговые права /srv/project/code: chmod 770, владелец alice:developers

Результаты проверок:
- bob (в группе developers) при правах 750 — НЕ МОГ создать файл,
  так как группе разрешено только r-x (чтение и выполнение, но не запись).
  После chmod 770 — смог создать файл (группа получила rwx).
- carol (не в группе developers) — НЕ МОГЛА войти в директорию,
  так как права для остальных (other) = --- (нет ни одного права).

Права файла q1.txt:
- Числовая запись: 640
  владелец=6(rw-), группа=4(r--), остальные=0(---)
- Символьная запись: -rw-r-----
  alice (владелец) — читает и пишет
  carol (группа auditors) — только читает
  bob и остальные — нет доступа


=== 3. ACL ===

Задача: дать carol доступ на чтение кода без изменения группы.
Команда: setfacl -m u:carol:r-x /srv/project/code

Чем ACL лучше стандартной модели:
Стандартная модель даёт права только трём субъектам: владелец, группа, остальные.
Чтобы дать carol доступ без ACL, пришлось бы добавить её в группу developers
(лишние права) или открыть other (небезопасно). ACL позволяет точечно выдать
права конкретному пользователю — это реализация принципа наименьших привилегий.


=== 4. sudo-политики ===

alice — разрешено всё (NOPASSWD:ALL), ограничений нет
bob   — разрешено только apt и apt-get
carol — разрешено только journalctl и cat /var/log/*

Почему alice использует NOPASSWD:
Alice — администратор, выполняет широкий круг задач с привилегиями.
Постоянный ввод пароля снижал бы эффективность работы.

Реализованный принцип безопасности:
Принцип наименьших привилегий (Least Privilege) — каждый пользователь
получает ровно столько прав, сколько нужно для его задач, и не больше.
Это снижает ущерб при компрометации учётной записи.


=== 5. PAM ===

Модуль pam_unix.so подключается через @include common-auth в /etc/pam.d/sudo.
Это основной модуль аутентификации — проверяет пароль по базе /etc/shadow.

Что означает "required":
Если модуль с типом required возвращает ошибку, аутентификация будет отклонена.
PAM продолжает выполнять остальные модули, чтобы не раскрывать место ошибки.

Текущие настройки (pam_unix.so):
- obscure   — запрещает слишком простые пароли
- yescrypt  — современный алгоритм хеширования (Debian 11+)

Модуль pam_pwquality не установлен.
Чтобы добавить минимальную длину пароля 12 символов:
  sudo apt install libpam-pwquality
Затем в /etc/pam.d/common-password добавить:
  password required pam_pwquality.so minlen=12


=== 6. Выводы ===

Изучены основные механизмы управления доступом в Linux. Стандартная модель
(chmod/chown) разграничивает доступ между владельцем, группой и остальными.
ACL расширяет модель, позволяя точечно настраивать права без изменения групп.
sudo реализует принцип наименьших привилегий. PAM обеспечивает модульную
аутентификацию и гибкую настройку политики паролей.
