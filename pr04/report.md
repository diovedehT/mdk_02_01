## 2. auditd — настройка правил

### Применённые правила (вывод auditctl -l)

    -w /etc/passwd -p rwa -k auth-files
    -w /etc/shadow -p rwa -k auth-files
    -w /etc/sudoers -p rwa -k priv-files
    -w /root -p rwa -k access-denied
    -a always,exit -F arch=b64 -S openat -F exit=-EACCES -k access-denied
    -a always,exit -F arch=b64 -S openat -F exit=-EPERM -k access-denied

### Типы записей в журнале auditd и что означает type=SYSCALL

В выводе sudo tail -20 /var/log/audit/audit.log видны следующие типы:

USER_ACCT  — проверка учётной записи пользователя PAM-системой
CRED_ACQ   — получение учётных данных
LOGIN      — событие входа пользователя в систему
SYSCALL    — системный вызов ядра
PROCTITLE  — полная командная строка процесса
USER_START — начало пользовательской сессии (PAM session open)
USER_LOGIN — успешный вход пользователя
USER_AUTH  — попытка аутентификации

type=SYSCALL фиксирует обращение программы к ядру ОС. Содержит архитектуру
(arch), номер системного вызова (syscall), результат (success=yes/no),
PID процесса, UID/GID пользователя, имя команды (comm) и путь к файлу (exe).
Это наиболее подробный тип записи — позволяет восстановить точное действие
на уровне ядра.

### Колонки в выводе aureport

#         — порядковый номер записи
date time — дата и время события
file      — путь к файлу
syscall   — номер системного вызова (257=openat, 59=execve и т.д.)
success   — результат (yes — успешно, no — отказ)
exe       — исполняемый файл, выполнивший вызов
auid      — Audit UID инициатора (сохраняется при su/sudo)
event     — номер события в журнале auditd

### Разбор записи SYSCALL

Пример из журнала:
    type=SYSCALL msg=audit(07.05.2026 22:34:53):
    arch=x86_64 syscall=openat success=no exit=EACCES
    auid=user uid=bob euid=bob comm=cat exe=/usr/bin/cat key=auth-files

| Поле | Значение     | Что означает                                                        |
|------|--------------|---------------------------------------------------------------------|
| auid | user (1000)  | Audit UID — кто вошёл изначально, не меняется при su/sudo          |
| uid  | bob (1001)   | Real UID — текущий UID процесса после su - bob                      |
| euid | bob (1001)   | Effective UID — с чьими правами выполняется действие                |
| comm | cat          | Краткое имя команды (до 15 символов)                                |
| exe  | /usr/bin/cat | Полный путь к исполняемому файлу                                    |

### Как auditd отличает чтение файла от его изменения

auditd различает операции по двум критериям:

1. Поле syscall:
   openat (257) — открытие файла (чтение или запись зависит от флагов)
   read (0)     — чтение данных
   write (1)    — запись данных
   unlink (87)  — удаление файла

2. Флаги открытия файла в поле a2:
   O_RDONLY            — только чтение
   O_WRONLY            — только запись
   O_WRONLY|O_CREAT    — создание и запись
   O_WRONLY|O_TRUNC    — перезапись

Пример:
   syscall=openat a2=O_RDONLY           -> чтение файла
   syscall=openat a2=O_WRONLY|O_CREAT  -> изменение файла

В правилах auditd флаги -p rwa: r — read, w — write, a — attribute change.

### Как auditd помог расследованию

auditd позволил точно установить кто, когда и к каким файлам пытался
получить доступ. Ключевую роль сыграло поле auid: несмотря на то что
действия выполнялись от имени bob (через su - bob), auid=user (1000)
однозначно указал на реального инициатора. Правила с ключами auth-files,
priv-files и access-denied позволили быстро отфильтровать нужные события
через ausearch -k.

### Чем auditd лучше journalctl для задач аудита безопасности

journalctl собирает только то, что приложения сами записывают в лог.
Злоумышленник может остановить службу логирования или удалить файлы
журналов. auditd работает на уровне ядра и перехватывает системные вызовы
напрямую — приложение не может скрыть свои действия. auditd фиксирует
auid, который сохраняется при смене пользователя через su/sudo, что
критично для установления реального нарушителя.

## Контрольные вопросы

1. Чем отличается journalctl от auditd?

journalctl — инструмент для чтения журнала systemd. Собирает логи от
служб и ядра, но работает на уровне приложений. auditd работает на уровне
ядра Linux и перехватывает системные вызовы напрямую — приложение не может
скрыть свои действия. journalctl удобен для общего мониторинга, auditd —
для глубокого аудита безопасности.

2. Что такое auid и почему он важен для расследования инцидентов?

auid (Audit User ID) — UID пользователя, вошедшего в систему изначально.
В отличие от uid, поле auid не меняется при переключении через su или sudo.
Это позволяет установить реального человека за любым действием, даже если
он работал под чужой учётной записью.

3. Что произойдёт с журналом auditd при перезагрузке системы?

Временные правила, добавленные через auditctl, будут утеряны. Для
сохранения нужно записать их в /etc/audit/rules.d/ и применить через
augenrules --load. Сам журнал /var/log/audit/audit.log сохраняется на
диске и не теряется при перезагрузке.

4. Как злоумышленник может скрыть действия от journalctl, но не от auditd?

journalctl читает логи, которые формируют сами приложения. Злоумышленник
может остановить службу логирования или удалить файлы журналов. auditd
работает на уровне ядра — его нельзя обойти без root-прав. Даже при
удалении файла лога ядро продолжит фиксировать системные вызовы пока
работает демон auditd.

5. Для чего используется флаг -k в правилах auditd?

Флаг -k задаёт ключ (метку) для правила. Ключ записывается в поле key
каждого события. Это позволяет быстро фильтровать события по теме:
ausearch -k watch-passwd покажет только события доступа к файлам паролей.

6. Что означает success=no в записи SYSCALL?

success=no означает что системный вызов завершился с ошибкой — операция
не выполнена. Поле exit указывает код ошибки:
EACCES — отказано в доступе (нет прав)
ENOENT — файл не найден
EEXIST — файл уже существует
Это позволяет отличить успешные операции от заблокированных попыток.
