# ПР №9. Следы вредоносного ПО в Linux

## 1. Что было посажено

| Механизм | Место | Команда/файл |
|----------|-------|-------------|
| Cron | crontab пользователя | /tmp/.hidden_malware/backdoor.sh |
| Systemd | ~/.config/systemd/user/ | system-helper.service |
| Shell profile | ~/.bashrc (строка 116) | /tmp/.hidden_malware/backdoor.sh & |
| Процесс | /tmp/.hidden_malware/ | listener.sh на порту 4444 |

## 2. Что нашли — процессы

Команда: `ps aux | grep '/tmp'`

Результат:
```
user  2961  0.0  0.1  6940  3168  pts/0  S  июн14  0:00  /bin/bash /tmp/.hidden_malware/listener.sh
user  2962  0.0  0.0  3208  1068  pts/0  S  июн14  0:00  nc -l -p 4444
```

Что подозрительно: процессы запущены из скрытой папки `/tmp/.hidden_malware/` — имя начинается с точки (скрытая директория), расположена в `/tmp` где легитимные программы постоянно не живут. `nc` (netcat) слушает входящие соединения на порту 4444 без какой-либо видимой причины — классический признак backdoor.

## 3. Что нашли — сетевые соединения

Команда: `ss -tulnp`

Подозрительный порт: **4444**

Процесс: nc, PID 2962

Команда: `sudo lsof -i :4444`

Результат:
```
COMMAND  PID  USER  FD   TYPE  DEVICE  SIZE/OFF  NODE  NAME
nc      2962  user   3u  IPv4   41930       0t0  TCP   *:4444 (LISTEN)
```

Как lsof связывает порт с процессом: lsof показывает какой процесс (по PID и имени) держит открытый сетевой сокет. В данном случае процесс `nc` с PID 2962 от пользователя `user` держит TCP-сокет на порту 4444 в состоянии LISTEN — то есть ждёт входящих подключений. Это позволяет однозначно связать подозрительный порт с конкретным исполняемым файлом.

## 4. Что нашли — автозапуск

### Cron

Команда: `crontab -l` / `sudo cat /var/spool/cron/crontabs/user`

```
@reboot /tmp/.hidden_malware/backdoor.sh &
*/5 * * * * /tmp/.hidden_malware/backdoor.sh &
```

Что подозрительно: две записи запускают скрипт из скрытой папки в `/tmp`. Запись `@reboot` обеспечивает выживание после перезагрузки, `*/5 * * * *` перезапускает скрипт каждые 5 минут — если процесс убить, через 5 минут он вернётся. Путь к скрипту в `/tmp/.hidden_malware/` — типичный признак вредоноса.

### Systemd

Команда: `systemctl --user list-unit-files --state=enabled --type=service`

Среди системных сервисов обнаружен подозрительный:
```
system-helper.service   enabled   enabled
```

Содержимое unit-файла (`~/.config/systemd/user/system-helper.service`):
```ini
[Unit]
Description=System Helper Service
After=default.target

[Service]
ExecStart=/tmp/.hidden_malware/backdoor.sh
Restart=always
RestartSec=10

[Install]
WantedBy=default.target
```

Статус: `activating (auto-restart)` — сервис пытается перезапуститься после каждого сбоя (backdoor.sh отсутствует в /tmp после перезагрузки ВМ, но сервис продолжает попытки).

Что подозрительно: сервис создан вручную в пользовательском каталоге `~/.config/systemd/user/` без установки через пакетный менеджер. `ExecStart` указывает на `/tmp/.hidden_malware/` — нелегитимное место. `Restart=always` обеспечивает автоматический перезапуск при любом сбое. Описание `System Helper Service` — намеренно общее для маскировки.

### ~/.bashrc

Команда: `grep -n "tmp\|hidden" ~/.bashrc`

Строка которую нашли:
```
/home/user/.bashrc:116:/tmp/.hidden_malware/backdoor.sh &
```

Где находится: строка 116, в конце файла после всех легитимных настроек. Выполняется при каждом открытии терминала пользователем.

## Выводы

В ходе практической работы была смоделирована компрометация системы Linux и проведено её расследование. Вредонос использовал четыре механизма закрепления: cron (перезапуск каждые 5 минут и при загрузке), пользовательский systemd-сервис с `Restart=always`, добавление в ~/.bashrc и запущенный процесс-слушатель на порту 4444.

Наиболее неочевидным местом оказался пользовательский systemd-сервис в `~/.config/systemd/user/` — он не требует прав root для создания и включения, при этом выживает после перезагрузки и автоматически перезапускается. Многие администраторы проверяют только системные сервисы в `/etc/systemd/system/`, упуская пользовательские.

Ключевой вывод: вредонос в Linux не обязательно требует прав root — достаточно доступа к учётной записи пользователя чтобы прописаться в cron, bashrc и пользовательский systemd, запустить слушатель на произвольном порту и сохранять присутствие в системе неограниченно долго.

## Ответы на контрольные вопросы

**1. Назовите пять мест в Linux где вредонос может прописать автозапуск. Чем они отличаются по уровню привилегий?**

- **crontab пользователя** (`crontab -e`) — не требует root, работает от имени пользователя
- **~/.bashrc / ~/.profile** — не требует root, выполняется при каждом входе пользователя в систему
- **~/.config/systemd/user/** — не требует root, пользовательский systemd-сервис выживает после перезагрузки
- **/etc/crontab и /etc/cron.d/** — требует root, выполняется от любого пользователя включая root
- **/etc/rc.local и /etc/systemd/system/** — требует root, запускается при загрузке системы с правами root

Первые три доступны без привилегий — скомпрометировав аккаунт обычного пользователя злоумышленник уже может закрепиться. Последние два требуют повышения привилегий но дают более широкие возможности.

**2. Чем lsof отличается от ss? Когда использовать каждый?**

`ss` показывает сетевые сокеты и соединения — быстро отвечает на вопрос "какие порты открыты и в каком состоянии". `lsof` показывает все открытые файлы процесса включая сокеты, обычные файлы, библиотеки, устройства — отвечает на вопрос "что именно делает этот процесс и какие ресурсы использует".

Используй `ss` когда нужно быстро проверить открытые порты и соединения. Используй `lsof -p PID` когда нужно детально изучить конкретный подозрительный процесс — какие файлы читает, куда пишет, какие соединения держит.

**3. Что означает строка (deleted) в выводе lsof? Почему вредоносы используют этот трюк?**

`(deleted)` означает что файл был удалён с диска, но процесс всё ещё держит его открытым — файл продолжает существовать в памяти пока процесс не закроет его дескриптор. Вредоносы используют этот трюк чтобы скрыть своё присутствие: запускают исполняемый файл, затем удаляют его с диска — при просмотре файловой системы файла уже нет, но процесс продолжает работать. Обнаружить такой процесс можно только через `lsof | grep deleted`.

**4. Пользовательский systemd-сервис в ~/.config/systemd/user/ — нужен ли root чтобы его создать? Почему это опасно?**

Root не нужен. Любой пользователь может создать unit-файл в своём `~/.config/systemd/user/` и включить его через `systemctl --user enable`. Это опасно потому что: злоумышленник получивший доступ к аккаунту обычного пользователя сразу получает механизм персистентности без необходимости повышать привилегии. Большинство администраторов проверяют только `/etc/systemd/system/` и упускают пользовательские сервисы. Сервис автоматически запускается при входе пользователя в систему и может иметь `Restart=always`.

**5. Вредонос переименовал себя в systemd-journald. Как отличить настоящий системный процесс от поддельного?**

Несколько способов проверки:
- `ls -la /proc/PID/exe` — показывает реальный путь к исполняемому файлу. Настоящий `systemd-journald` находится в `/lib/systemd/`, поддельный — в `/tmp/` или домашней папке
- `stat /proc/PID/exe` — время создания файла. Системный процесс создан при установке ОС, поддельный — недавно
- `lsof -p PID` — настоящий системный процесс открывает системные файлы журналов, поддельный может не открывать ничего или открывать файлы из /tmp
- `systemctl status имя` — настоящий сервис управляется systemd и имеет корректный unit-файл в `/lib/systemd/system/`
